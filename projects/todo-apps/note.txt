# models.py(Database Architecture):
[code]:
    owner =models.ForeignKey(user, on_delete = models.CASCADE, ....)

Single registered user can create many task in another word it tell the Onet to Many Relationship.
and on_delete=models.CASCADE means if a user deletes their account, all of their task are automatically wiped from the database to prevent orphaned data.

[Title, description, is_completed, due_date]:
- These are the main attribute of this apps, blank=True,null=True use for description and due date becaude user aren't forces to add a description or a deadline if they don't want to.

[PRIORITY_CHOICES & priority = models.IntegerField(...)]

- So, instead of used ASCII character i used number character because if we use L,H,M the ASCII values make some problem to sort the task,
it stores number 1,2 and 3 in the database to optimize space and calculation speed, but display human-readable word like 'High', 'Medium', 'Low' in the interface.


[created_at = models.DateTimeField(auto_now_add=True)]
- "The auto_now_add=True argument makes Django automatically timestamp the exact date and time a task is created, preventing users from altering this value."

class Meta: ordering = [
    'is_completed', 'priority', 'due_date'
    ]

This is the default sorting engine for my data. It ensures that uncompleted tasks stay at the top. If two tasks have the same completion status, it ranks them by priority (1 before 2).
If priority matches, it orders them chronologically by their upcoming deadline."

Explanation: "This is a Python magic method. Instead of showing an ugly object pointer like <TodoItem object>, it forces Django to read out the actual string value of the task title in the logs and admin interface."



3. serilizers.py [Data Translator]

class TodoItemSerializer(serializers.ModelSerializer):
    class Meta:
        model = TodoItem
        fields = '__all__'

** class TOdoItemSerializer(serializers.ModelSerializer): just telling Django Rest Framework to look at a Django Database model to automatically figure out what fields the serializer should have. This saves you from having to rewrite all model by hand.

** class meta: This is an inner configuration class used by Django and DRF to provide metadata to the parent class. It configures how the TodoItemSerializer should behave.

** model = TodoItem: This tells the serializer which Django model it is mirroring. In this case, it is linked to a model named TodoItem. The serializer will now look at that model's database schema to understand its data types, validation rules, and relationships.

** fields = '__all__': This is a shortcut that tells DRF: took every single field form the TodoItem model and include it in the API serialization. 

4. views.py(): 

class TodoItemViewSet(viewsets.ModelViewSet):
    queryset = TodoItem.objects.all()
    serializer_class = TodoItemSerializer

if Serializer is the translator of API, the ViewSet is the Traffic controller. by using DRF ModelViewSet it works manually write individual functions for viewing, editing or deleting items. This single viewset automatically hooks up all five standard REST actions: List, CREate, Retrive, Update, & Destroy using the specified model queryset and translator serializer.

6. Admin.py:

@admin.register(TodoItem)
class TodoItemAdmin(admin.ModelAdmin):
    list_display = ('title', 'category', 'priority', 'due_date', 'is_completed', 'is_overdue')
    list_display_links = ('title',)
    list_filter = ('is_completed', 'priority', 'category')
    search_fields = ('title', 'description', 'category')
    list_editable = ('is_completed', 'priority')

This decorates the built-in Django Admin view. list_display selects my data columns, list_filter creates sidebar filtering widgets, and search_fields lets me query specific keywords instantly. list_editable lets me change a task's completion or priority status directly from the main summary table list without clicking inside the object profile page."
